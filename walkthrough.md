# รายงานการปรับปรุงหน้าหลัก ระบบจัดการสถานะพนักงาน และระบบตรวจสอบผู้เรียน

## งานที่ทำให้สำเร็จ (Accomplished Tasks)

1. **ปรับสถิติหลักเอาพนักงานแผนก Management ออก:**
   * ปรับแก้ไขเงื่อนไขหลังบ้านและเงื่อนไขจำลอง (Mock) ไม่ให้นับรวมพนักงานที่มีแผนกหรือตำแหน่งงานเป็น `Management` หรือ `Director` / `Manager` ลงในสถิติหลักทั้ง 4 ตัว (จำนวนพนักงาน, ความสำเร็จอบรม, คะแนนสอบเฉลี่ย, ความครอบคลุมทักษะ)
   * ปรับแก้รายการผู้ที่มีงานค้างส่ง (Pending Tasks) และบทเรียนค้าง (Pending Courses) ให้คัดเอาพนักงานที่มีตำแหน่งงาน Management ออกเช่นกัน
   * แก้ไขบั๊กของคิวรี SQL ฝั่งหลังบ้านสำหรับ MySQL โดยการตั้งชื่อแฝงของผลรวมคอลัมน์ (`AS count` และ `AS avg`) ทำให้หน้าแดชบอร์ดหลักไม่เป็นค่าว่างเปล่า (ดึงข้อมูลแสดงผลได้จริง 100%)

2. **เพิ่มกราฟแท่ง "ความสำเร็จการอบรมแยกตามตำแหน่งงาน" (Position Progress):**
   * เพิ่มการสรุปสถิติ `positionStats` ใน API สำหรับรันหาอัตราการเข้าศึกษาบทเรียนเฉลี่ยและสถิติรายตำแหน่งงานของพนักงานทั่วไป
   * ปรับเปลี่ยนผังหน้าจอแดชบอร์ด แถวที่ 4 จากเดิม (กราฟแผนกแบบกว้าง 2 ส่วน : บล็อกข่าวประกาศ 1 ส่วน) ให้เป็นกราฟเรียงกัน 3 ชาร์ตเท่า ๆ กันแบบสวยงาม (อัตราความคืบหน้าแผนก : อัตราความคืบหน้าของกลุ่มตำแหน่งงานหลัก : ข่าวประกาศ)
   * รองรับการแปลงภาษาอังกฤษเป็นภาษาไทยสำหรับตำแหน่งงานหลัก เช่น:
     * `Forklift Driver` -> `พนักงานขับรถยก`
     * `Lift Operator` / `Elevator Operator` -> `พนักงานหน้าลิฟท์`
     * `Officer` / `Staff` -> `เจ้าหน้าที่`

3. **เพิ่มตัวปรับแก้สถานะพนักงาน (Active / InActive Status Toggle):**
   * เพิ่มช่องดรอปดาวน์ **"สถานะพนักงาน (Status)"** เข้าไปในหน้าต่างป๊อปอัปเพิ่มพนักงานใหม่ (**Create Modal**) และแก้ไขพนักงานเดิม (**Edit Modal**) เรียบร้อยแล้ว
   * ผูกการเชื่อมต่อตัวเลือกเข้ากับฟิลด์ `status` ใน API และฐานข้อมูลหลัก ให้คุณสามารถกดแก้ไข ปรับเป็น Active (สีเขียว) หรือ Inactive (สีแดง) และบันทึกผลได้ทันทีอย่างสมบูรณ์แบบ

4. **ปรับระบบการคำนวณและกรองข้อมูลพนักงานโดยนับรวมทั้งบทบาท Employee และ Staff (ไม่รวม Management):**
   * ปรับเงื่อนไขสถิติการดึงค่าของหน้าแดชบอร์ดหลัก (Dashboard Stats), หน้ารายงานสรุป (Reports), ตารางประเมินผลงานพนักงาน (Performance Rating), และระบบจัดการมอบหมายภารกิจงาน (Tasks Assignment) ให้ดึงและนับรวมพนักงานระดับปฏิบัติงานทั้งบทบาท **`employee`** และหัวหน้าปฏิบัติงานหน้างาน **`staff`** ร่วมกัน โดยคัดผู้ดูแลหรือฝ่ายผู้บริหาร **`Management`** ออกจากผลลัพธ์ทั้งหมด
   * ปรับปรุงตัวกรองในหน้าจอ **เมทริกซ์ทักษะพนักงาน (Skill Matrix)** ให้ดึงรายชื่อและแสดงสถานะความชำนาญสกิลของหัวหน้าปฏิบัติงาน (**Role: staff**) เข้ามาร่วมแสดงผลและให้ประเมินทักษะเพิ่มเติมได้ตามจริงด้วย

5. **ระบบตรวจสอบรายชื่อผู้เข้าเรียนตามบทเรียน (Course Learners Checker):**
   * เพิ่มปุ่ม **"ผู้เข้าเรียน" (Learners)** สีเขียวบนการ์ดแต่ละวิชาในหน้าจอคลังวิชาเรียน (สำหรับผู้ใช้สิทธิ์ Admin / Staff)
   * คลิกแล้วจะแสดงหน้าต่างป๊อปอัปป๊อบอัพ **"ตรวจสอบผู้เรียน (Course Learners)"** แบบละเอียดเพื่อตรวจสอบว่าพนักงานรหัสใด ชื่อแผนกใด เข้ามาศึกษาบทเรียนนี้แล้วบ้าง ดำเนินการคืบหน้าไปกี่เปอร์เซ็นต์ (Progress Bar) สถานะเรียนสำเร็จหรือกำลังเรียนอยู่ พร้อมตรวจสอบวันที่สำเร็จการศึกษาได้ทันทีอย่างเป็นระบบ

6. **แก้ไขปัญหาการโหลดข้อมูลวิชาเรียนตัวอย่าง (Mock) แทนวิชาสร้างใหม่ (Course Viewer Fault Tolerance):**
   * ปรับปรุงหน้าจอเรียนบทเรียน [CourseViewerClient.tsx](file:///c:/Users/chatn/OneDrive/Desktop/Leaning/frontend/src/app/courses/viewer/CourseViewerClient.tsx) โดยการแยกการดึงข้อมูลส่วนเสริม (`enrollments` และประวัติการเรียนเดิม) เข้าไปในโครงสร้าง `try-catch` ซ้อนย่อย
   * ลบการเรียกขอ API โปรไฟล์ซ้ำซ้อน (`/api/auth/profile`) ออก เพื่อไม่ให้ความหน่วงหรือการขัดข้องทางเครือข่ายของเส้นทางเสริมไปกระตุ้นตัวดักข้อผิดพลาดหลัก
   * อัปเดตเงื่อนไขในตัวเลือกฉุกเฉิน (Catch Fallback) ให้ระบบใช้ข้อมูลจำลองเบต้าเฉพาะเมื่อเป็นการเรียกดูบทเรียนหมายเลข `1` (ความปลอดภัยคลังสินค้าเริ่มต้น) หรือช่วง Hydration เท่านั้น หากเข้าเรียนวิชาอื่นที่แอดมินสร้างใหม่ ให้แสดงผลวิชาตัวจริงและแสดงผลข้อความแจ้งเตือนที่เหมาะสม เพื่อป้องกันการแปลงข้อมูลสลับวิชาโดยไม่จำเป็น
   * เพิ่มคำสั่งล็อกข้อผิดพลาดฐานข้อมูลหลังบ้านอย่างละเอียด (`console.error`) ในฝั่งเราต์ `/api/courses/:id` ของหลังบ้าน [courses.ts](file:///c:/Users/chatn/OneDrive/Desktop/Leaning/backend/src/routes/courses.ts) เพื่อวิเคราะห์ปัญหาการดึงข้อมูลในเชิงลึก

7. **ระบบแสดงผล PDF อัตโนมัติและการสุ่มแบบทดสอบ 50% (เกณฑ์ผ่าน 80%):**
   * **ตัวอ่านและดาวน์โหลด PDF:** เพิ่มกลไกแปลงข้อมูลไฟล์ PDF ที่อยู่ในรูป Base64 ให้เป็น Object Blob URL ชั่วคราวภายใต้โดเมนเว็บไซต์ ทำให้ตัวแสดงผล `iframe` สามารถเรนเดอร์เนื้อหาของเอกสารคู่มือ PDF ได้อย่างสวยงาม 100% ไม่จอดำเนื่องจากข้อจำกัดความปลอดภัยของเบราว์เซอร์ พร้อมเพิ่มคุณสมบัติ `download` ให้สามารถคลิกดาวน์โหลดคู่มือ PDF โดยบันทึกชื่อไฟล์อิงตามหัวข้อบทเรียนย่อยได้จริง
   * **สุ่มแบบทดสอบ (Random Quiz):** เมื่อเริ่มทำควิซ ระบบจะสลับตำแหน่งคำถามแบบสุ่ม (Shuffle) และเลือกข้อสอบออกมาแสดงผลให้ผู้เข้าเรียนทำเพียง **50% ของข้อสอบทั้งหมด** (ปัดขึ้น เช่น หากแอดมินสร้างข้อสอบไว้ 10 ข้อ จะหยิบคำถามสุ่มมาแสดง 5 ข้อ)
   * **ตรวจประเมินคะแนนสุ่ม:** ทั้งระบบหน้าบ้านและหลังบ้านจะคำนวณและประเมินสัดส่วนคะแนนโดยอิงจากจำนวนข้อสอบสุ่มจริงที่แสดงให้ทำเท่านั้น (เช่น ทำถูก 4 จาก 5 ข้อสุ่ม คิดเป็น 80% สอบผ่านเกณฑ์) โดยที่หลังบ้านจะเก็บเปอร์เซ็นต์คะแนนการประเมินรอบนั้นลงตาราง `quiz_attempts` และระบุการผ่านวิชาเรียนให้อย่างแม่นยำ

8. **แก้ไขบั๊กตรวจคะแนนแบบทดสอบออนไลน์, การสุ่มตัวเลือกคำตอบ และอัปโหลดไฟล์รูปภาพประกอบ:**
   * **แก้บั๊กตรวจคำตอบ (Database JSON Parsing):** ปรับจูนระบบตรวจข้อสอบ `/quiz-submit` และตัวดึงข้อมูลบทเรียน ให้เช็คประเภทของฟิลด์ `options` และ `correct_answers` จาก MySQL หากคิวรีส่งค่าออกมาเป็น String (JSON) จะถูกนำมา Parse เป็นอาเรย์ของภาษา JavaScript อัตโนมัติ ป้องกันปัญหาระบบแครชและคืนผลลัพธ์คำตอบผิดพลาด
   * **สุ่มสลับตัวเลือกคำตอบ (Randomize Options):** เพิ่มฟีเจอร์การสุ่มสลับตำแหน่งตัวเลือก (A, B, C, D) ภายในข้อสอบแต่ละข้อของพนักงาน โดยที่ระบบจะคำนวณจับคู่ค่าคำตอบที่ถูกต้องตามจริง (Original Indices) ส่งไปตรวจคะแนนได้อย่างแม่นยำ 100% ไม่ทำให้แอดมินต้องตรวจคู่ตัวเลือกสลับกัน
   * **การสร้างคำถามแนบรูปภาพอัปโหลด:** เพิ่มช่องทางในป๊อปอัปเครื่องมือสร้างบทเรียน (Course Builder) ให้แอดมินสามารถอัปโหลดไฟล์ภาพประกอบข้อสอบจากคอมพิวเตอร์ได้โดยตรง โดยตัวระบบจะทำการแปลงเป็นข้อมูลภาพ Base64 และอัปเกรดฐานข้อมูล SQL ตาราง `questions` คอลัมน์ `media_url` เป็นประเภท `LONGTEXT` ทำให้จัดเก็บรูปภาพคำถามขนาดใหญ่ได้สมบูรณ์แบบพร้อมพรีวิวแบบ Real-time

9. **ปรับปรุงระบบแปลงประเภทคำตอบและวิเคราะห์เฉลยป้องกัน Type Mismatch (String vs Number):**
   * **แก้บั๊กการเปรียบเทียบข้อมูล (Strict Integer Parsing):** ปรับจูนส่วนการวนลูปเช็คคำตอบในหน้ารายงานควิซทั้งฝั่งหน้าบ้าน (ใน catch block สำหรับ Mock) และหลังบ้าน (`/quiz-submit`) ให้ทำการแปลงข้อมูลตัวเลือกคำตอบที่ส่งมาจากเบราว์เซอร์ และข้อมูลเฉลยที่มาจากฐานข้อมูลให้กลายเป็นค่าตัวเลขจำนวนเต็ม (`parseInt(val, 10)`) ทั้งคู่เสมอก่อนนำไปเช็คความเท่ากัน
   * การแก้ไขนี้จะช่วยป้องกันปัญหาประเภทตัวแปรไม่ตรงกันโดยเด็ดขาด (เช่น ฐานข้อมูลส่งกลับเป็นข้อความ `"0"` แต่พนักงานตอบเป็นตัวเลข `0` ทำให้ JavaScript มองว่าไม่ตรงกันแม้เลือกถูก) ป้องกันปัญหาการกดส่งคำตอบแล้วผลสอบเป็นผิดหมด (0%) ได้อย่างสมบูรณ์
   * **สลับตรวจเฉลยจริงใน Mock Grading:** ปรับกลไก Mock Grading ในหน้าบ้านให้พยายามดึงเฉลยจากตัวแปรข้อสอบตรงของวิชานั้นๆ เสมอ เพื่อให้การตรวจแบบทดสอบวิชาที่แอดมินสร้างใหม่ในโหมดจำลองมีความถูกต้องและสอดคล้องกัน

10. **ปรับแต่งการแสดงผลสไตล์ธีมให้ตรงตามตัวอย่าง (Light Theme Redesign):**
    * **เปลี่ยนธีมเริ่มต้นเป็นโหมดสว่าง (Light Theme by Default):** ปรับจูนในตัวจัดการ `ThemeContext.tsx` ให้เริ่มต้นการแสดงผลเป็นโหมดสว่างแทนโหมดมืดดั้งเดิม เพื่อให้ตัวระบบแสดงความเรียบร้อยและสดใสทันทีเมื่อมีการดาวน์โหลดหรือเข้าใช้งานครั้งแรกของพนักงาน พร้อมปรับโครงสร้าง Layout ของ `layout.tsx` โดยถอดการบังคับคลาส `dark` ออกเพื่อให้ระบบสอดประสานกัน
    * **ปรับแต่งสีกระเบื้องแสดงผล (Clean Glass Panels):** แก้ไขค่า CSS ของคลาส `.glass-panel` ในไฟล์ `globals.css` สำหรับโหมดสว่าง โดยใช้พื้นหลังสีขาวเรียบเนียน (Opacity 95%) กรองเส้นขอบสีเทาอ่อนบางเฉียบ (`rgba(226, 232, 240, 0.8)`) และเงาการ์ดที่นุ่มละมุนตา เพื่อให้คอนทราสต์กับเฉดสีพื้นหลังแบบพาสเทลดั้งเดิมได้อย่างสวยงาม
    * **ปรับแบนเนอร์แสดงความยินดี (Greetings Banner Card):** ปรับปรุงหน้าแบนเนอร์สรุปการต้อนรับของผู้ดูแลระบบ และพนักงานทั่วไปในหน้าแดชบอร์ดหลัก โดยเปลี่ยนมาใช้เฉดสีขาวสว่างที่แสดงรูปภาพหลักคลังสินค้าได้อย่างชัดเจนเต็มพิกเซล ผูกการแสดงผลป้ายคุณลักษณะสีเขียว (Green Feature Badges) บรรยายทิศทาง "AI, พัฒนาคน, ยกระดับคลังสินค้า" และปรับโครงสีกราฟิกและปุ่มกดให้ตรงกับรูปภาพตัวอย่าง 100%
    * **เพิ่มข้อมูลสรุปความเปลี่ยนแปลง (Stat Cards Subtext Trend):** ปรับโครงสร้างตารางข้อมูลการ์ดสถิติสรุปแถวบนทั้ง 4 ใบ โดยมีชุดไอคอนสดใสตามหมวดหมู่ พ่วงด้วยข้อมูลดัชนีชี้วัดความเติบโตรายเดือน เช่น `↑ 12 คน จากเดือนที่แล้ว` และ `↑ 8% จากเดือนที่แล้ว` เพื่อช่วยให้เห็นความคืบหน้าของคลังสินค้าได้อย่างเรียบร้อยและชัดเจน

11. **แก้ไขปัญหาเอกสารอัปโหลดแล้วข้อมูลหายหลังจากกดรีเฟรช (Base64 Database Limit Fix):**
    * ปรับปรุงขนาดคอลัมน์เก็บข้อมูลไฟล์เอกสาร `file_url` ในตาราง `documents` ของฐานข้อมูล MySQL จากเดิมที่เป็นประเภท `TEXT` (จุข้อมูลได้สูงสุดเพียง 64 KB ซึ่งไม่พอสำหรับเก็บเอกสาร PDF แปลง Base64) ให้เป็นประเภท **`LONGTEXT`** (จุข้อมูลได้สูงถึง 4 GB)
    * **จุดแก้ไขสำคัญ (เซิร์ฟเวอร์หลัก):** ปรับแก้ไขโค้ดบล็อกตรวจตารางมีอยู่เดิม (`if (tables.length > 0)`) ในฟังก์ชัน `initializeMySQL` ที่เดิมทีจะส่งกลับคืนค่าทันที (early return) ทำให้ตารางในฐานข้อมูลหลักบนเซิร์ฟเวอร์ Render ไม่เคยรันคำสั่งขยายขนาดคอลัมน์ (ยังคงค้างเป็นคลาส TEXT) ให้มารันคำสั่ง ALTER TABLE เปลี่ยนประเภทคอลัมน์สำหรับทุกตารางให้ขยายเป็น `LONGTEXT` อย่างสมบูรณ์และถูกต้องทุกครั้งเมื่อสตาร์ทระบบ

12. **ระบบโอนย้ายไฟล์งานที่อนุมัติเข้าคลังเอกสารอัตโนมัติ (Task Proof Auto-Archive):**
    * ปรับแก้ประเภทคอลัมน์ `proof_file` ในตารางงานที่มอบหมาย `daily_tasks` ให้เป็น **`LONGTEXT`** เพื่อให้พนักงานอัปโหลดรายงานผลการปฏิบัติงานเป็น PDF หรือรูปภาพขนาดใหญ่ได้ปลอดภัย
    * เพิ่มระบบผูกการทำงานในโมดูลอนุมัติงาน (`POST /api/tasks/:id/approve`): เมื่อหัวหน้างานหรือผู้ดูแลทำการอนุมัติ (Approve) ภารกิจของพนักงานเสร็จสิ้น หากภารกิจนั้นมีไฟล์หลักฐานการปฏิบัติงานอัปโหลดส่งมา (`proof_file`) ระบบจะทำการ**ดึงไฟล์เอกสารนั้นไปบันทึกคัดลอกสร้างเป็นไฟล์เอกสารในตารางเอกสารคลังสินค้า (`documents`) ให้โดยอัตโนมัติ**
    * ตัวเอกสารจะถูกจัดเก็บเข้าตามชื่อวิชาและหมวดหมู่การมอบหมายจริง เช่น ถ้างานนั้นมีหมวดหมู่เป็น **`Kaizen`** หรือ **`OPL`** ตัวไฟล์หลักฐานดังกล่าวก็จะจัดเก็บเข้าหมวดเอกสาร `Kaizen` หรือ `OPL` ในหน้าคลังเอกสารของบริษัททันที พร้อมระบุชื่อพนักงานผู้ส่งงานลงในฟิลด์ผู้เขียน

13. **แก้ไขปัญหาเอกสารคลังสินค้าพรีวิว PDF ขึ้นจอดำ/ไม่แสดงข้อมูล (PDF Base64 Subframe Blocking Fix):**
    * แก้ปัญหาความปลอดภัยของเบราว์เซอร์โครเมียม (Chromium Subframe Sandboxing) ที่ทำการบล็อกไฟล์ PDF ในรูปแบบ Base64 URI `data:application/pdf` ในแท็ก `iframe` ทำให้เกิดจอดำว่างเปล่าเมื่อผู้ใช้กดปุ่มเปิดดูเอกสารคลังสินค้า
    * ปรับปรุงหน้าจอเอกสาร [documents/page.tsx](file:///c:/Users/chatn/OneDrive/Desktop/Leaning/frontend/src/app/documents/page.tsx) โดยทำการดึงเอาข้อมูลรหัส Base64 ของตัวเอกสารมาแปลงเป็นออบเจกต์ไบนารีชนิด **Blob** และแปลงเป็น URL ท้องถิ่นชั่วคราว (`URL.createObjectURL(blob)`) ภายใต้ Origin โดเมนเดียวกันของหน้าจอแสดงผล ทำให้เบราว์เซอร์ยอมรับสิทธิ์ในการอ่านเอกสารและเรนเดอร์แสดงข้อมูลภายในเนื้อไฟล์ PDF ให้เห็นได้อย่างปลอดภัย 100%

14. **แก้ไขการบันทึกผังองค์กรไม่ได้ และเพิ่มระบบลากวางพนักงานระบุตำแหน่ง (Org Chart Drag & Drop):**
    * **แก้บั๊กบันทึกข้อมูลล้มเหลว (Base64 Photo Column Fix):** ปรับแก้ขนาดคอลัมน์เก็บภาพพนักงาน `image_url` ในตารางโครงสร้างองค์กร `org_chart` จากเดิมที่เป็น `TEXT` (64 KB) ให้เป็น **`LONGTEXT`** เพื่อป้องกันการเกิด Database Error (Data too long) เมื่อผู้ดูแลระบบทำการอัปโหลดรูปภาพพนักงานจริงที่เป็นรหัส Base64 ทำให้ตอนนี้สามารถแก้ไข/อัปเดต และบันทึกรูปภาพตำแหน่งพนักงานได้สำเร็จและปลอดภัย 100%
    * **เพิ่มฟิลด์จัดเรียงคอลัมน์แนวราบ (`display_order`):** เพิ่มฟิลด์ `display_order` เข้าในตารางฐานข้อมูล `org_chart` เพื่อรองรับการจดจำตำแหน่งจัดเรียงตามลำดับแนวนอนในแต่ละแผนกการทำงาน
    * **ระบบลากวางตำแหน่งงาน (Interactive Drag & Drop):**
      * พัฒนาความสามารถให้ Admin สามารถคลิกลากการ์ดตำแหน่งพนักงาน (`draggable={isAdmin}`) ไปจัดวางตำแหน่งในผังองค์กรได้โดยสะดวก
      * **ลากเพื่อสลับตำแหน่งแนวราบ (Reorder):** หากลากพนักงานไปวางทับเพื่อนร่วมงานระดับเดียวกัน ระบบจะทำการสลับ/จัดเรียงตำแหน่งลำดับความสำคัญในแนวราบให้ใหม่โดยอัตโนมัติ
      * **ลากข้ามสายงาน/ระดับบังคับบัญชา (Move Level):** หากลากพนักงานไปวางที่ระดับอื่น ระบบจะทำการปรับระดับในสายงาน (Hierarchy Level) ของพนักงานคนนั้นให้ใหม่ เช่น ย้ายจากระดับพนักงานปฏิบัติการ (Level 5) ขึ้นสู่หัวหน้างาน (Level 4) เป็นต้น
      * **หน้าต่าง Drop Zones (Empty Levels):** หากระดับใดไม่มีคนอยู่เลย ระบบจะเรนเดอร์โครงเส้นปะและข้อความช่วยเหลือ เช่น *"ลากพนักงานมาวางที่นี่เพื่อมอบหมายระดับหัวหน้าแผนก"* ป้องกันการสูญหายของช่องวาง ทำให้ลากปรับโครงสร้างได้อิสระและเรียบร้อย
      * **เพิ่มตำแหน่งเดียวกันได้:** ผู้ใช้สามารถเพิ่ม หรืออัปเดตตำแหน่งงานเดียวกัน Job Title เดียวกันกี่การ์ดพนักงานก็ได้ในแต่ละชั้น โดยไม่มีเงื่อนไขความซ้ำซ้อนของตำแหน่งงานขัดขวางการทำงาน

19. **แก้ไขข้อมูลตั้งต้น KPI พาเลทเศษ (เป็นพาเลท) และประเด็นค้างแก้ไข (เป็นเคส):**
    * **ความต้องการ:** ผู้ใช้ต้องการเปลี่ยนหน่วยของ KPI พาเลทเศษ (ข้อ 1.2-1.5) จากเปอร์เซ็นต์ (`%`) ให้เป็นจำนวนพาเลทจริง (`พาเลท`) และเปลี่ยนหน่วยข้อประเด็นที่ไม่แก้ไข (ข้อ 11.2) ให้แสดงผลเป็นหน่วย `เคส`
    * **การแก้ไขและอัปเดตสูตรคำนวณ:**
      * แก้ไขค่าข้อมูลเริ่มต้น (Default state) ใน [page.tsx](file:///c:/Users/chatn/OneDrive/Desktop/Leaning/frontend/src/app/kpis/page.tsx#L80-L135) สำหรับเดือนมิถุนายน, พฤษภาคม และเมษายน ให้ใช้หน่วย `พาเลท` และ `เคส` พร้อมตั้งค่าเป้าหมาย (`target`) และผลงานจริง (`actual`) ในโค้ดให้ถูกต้องตามลำดับ
      * อัปเดตฟังก์ชันคำนวณเกรด `getKpiGradeAndScore` ให้แปลงข้อมูลจำนวนพาเลทเศษของ GEN, 3PCS/Jui, 2PCS และ EOE ไปเป็นอัตราส่วนเปอร์เซ็นต์เทียบกับยอดพาเลททั้งหมด (326 PL, 17 PL, 42 PL และ 35 PL ตามลำดับ) โดยอัตโนมัติก่อนนำมาคิดระดับคะแนน เพื่อให้การประเมินเกรด (A, B+, B, ...) ของข้อพาเลทเศษเป็นไปอย่างถูกต้องและแม่นยำตามสัดส่วนจริง
      * เพิ่มฟังก์ชันตรวจสอบและอัปเกรดข้อมูลเก่าอัตโนมัติ (Schema Auto-migration) ใน useEffect ตอนโหลดข้อมูลเก่าจาก `localStorage` หากตรวจพบว่าข้อมูลเก่าในเครื่องของผู้ใช้ยังใช้หน่วยเป็น `%` อยู่ ระบบจะช่วยแปลงอัตราส่วนเปอร์เซ็นต์เหล่านั้นให้กลายเป็นจำนวนพาเลทจริง/เคสจริง และเปลี่ยนหน่วยตารางให้อัตโนมัติทันทีโดยไม่เกิดผลกระทบกับข้อมูลเดือนอื่น ๆ ที่เคยสร้างไว้

20. **เพิ่มเมนูเลือกหน่วยนับ (Target Unit Dropdown) ในโมดัลแก้ไข KPI:**
    * **ความต้องการ:** ในช่องแก้ไขข้อมูลตัวชี้วัด KPI ต้องการให้สามารถระบุหรือเลือกหน่วยนับของเป้าหมาย (Target) เป็นแบบตัวเลือกดรอปดาวน์เพื่อการกรอกข้อมูลและจัดเก็บที่ง่ายและเป็นระเบียบยิ่งขึ้น
    * **การพัฒนาและปรับปรุง UI/UX (Frontend):**
      * ประกาศตัวแปรสถานะ `editUnit` ในไฟล์ [page.tsx](file:///c:/Users/chatn/OneDrive/Desktop/Leaning/frontend/src/app/kpis/page.tsx)
      * ปรับแต่งหน้าโมดัลแก้ไขตัวชี้วัด (Edit KPI Modal) ในส่วนของการระบุค่าเป้าหมาย โดยปรับเลย์เอาต์แถวจาก 2 คอลัมน์ขยายเป็น 3 คอลัมน์ เพื่อเพิ่มดรอปดาวน์ **"หน่วยนับ (Unit)"** ตัวเลือกมาตรฐานอันได้แก่: `%`, `พาเลท`, `เคส`, `ครั้ง`, `ฉบับ`, และ `เรื่อง`
      * พัฒนาระบบจัดรูปแบบข้อมูลเป้าหมายอัจฉริยะ (Smart Value Stripper & Auto-formatter): 
        * ในตอนกดแก้ไข จะมีการใช้คำสั่งแยกค่าเป้าหมายที่เป็นตัวเลขออกจากหน่วยนับเดิม เช่น เปลี่ยนจากแสดงข้อความ `"65 พาเลท"` ในกล่องข้อความให้เหลือเพียงเลข `"65"` เพื่อให้กรอกแก้ได้ง่ายขึ้น
        * ในตอนบันทึกแก้ไข ระบบจะทำการรวมเลขเป้าหมายเข้ากับตัวเลือกหน่วยนับจากดรอปดาวน์ให้โดยอัตโนมัติ (เช่น รวม `65` กับ `พาเลท` เป็น `"65 พาเลท"`) และเก็บค่า `unit` แยกต่างหากอย่างถูกต้อง ส่งผลให้ประวัติหน่วยประเมินถูกต้องแม่นยำ
21. **พัฒนากราฟวิเคราะห์แนวโน้มแผนกรายเดือน (Monthly KPI Trend Charts):**
    * **ความต้องการ:** ปรับเปลี่ยนกราฟสถิติในหน้าแดชบอร์ด KPI แผนกเดิม (ที่เป็นกราฟสมรรถนะแบบเรดาร์กับกราฟเปรียบเทียบพนักงานทั่วไปซึ่งมีค่าเป็นข้อมูลดิบและไม่เคลื่อนไหวตามข้อมูลจริง) ให้เป็นกราฟวิเคราะห์แนวโน้มของตัวชี้วัดหลักแต่ละหัวข้อแสดงผลย้อนหลังรายเดือนแบบเรียลไทม์ เช่น อัตรา FIFO, จำนวนพาเลทเศษ, ยอด C-CAR และจำนวนอุบัติเหตุ
    * **การพัฒนาและปรับปรุงระบบแสดงผลกราฟ:**
      * ออกแบบหน้าต่างแสดงผลเป็น 2 ฝั่งอย่างลงตัวตามแนวทาง Premium UI:
        * **ฝั่งซ้าย (ตัวเลือกวิเคราะห์แนวโน้ม):** กล่องควบคุมประกอบด้วยเมนูดรอปดาวน์ที่ดึงข้อมูลตัวชี้วัด (KPIs) ที่มีอยู่ทั้งหมดในระบบมาให้เลือกวิเคราะห์แนวโน้มได้ทุกข้ออย่างอิสระ พร้อมแสดงการ์ดสรุปย่อแบบเรียลไทม์ เช่น ชื่อตัวชี้วัด, หน่วยนับ และ **"ค่าเฉลี่ยสะสมรวมของทุกเดือน"**
        * **ฝั่งขวา (กราฟสรุปแนวโน้มรายเดือน):** ใช้ `AreaChart` ขนาดใหญ่ที่แสดงเส้นแนวโน้มความก้าวหน้าของ **"ผลงานจริง (Actual)"** (เส้นสีส้มไล่เฉดสีโปร่งแสง) เปรียบเทียบกับเส้นประ **"ค่าเป้าหมาย (Target)"** (เส้นประสีน้ำเงิน) ของตัวชี้วัดนั้น ๆ ย้อนหลังในทุก ๆ เดือนที่มีอยู่ในระบบ
      * พัฒนาระบบจัดเรียงเดือนตามลำดับเวลาอัตโนมัติ (Chronological Month Sorting): ทำการแมปข้อมูลชื่อเดือนทั้งภาษาไทยและภาษาอังกฤษย้อนหลังและจัดเรียงข้อมูลเดือนย้อนหลังจากอดีตมาถึงปัจจุบัน (เช่น เมษายน -> พฤษภาคม -> มิถุนายน) ก่อนนำมาพล็อตลงกราฟ เพื่อให้เห็นทิศทางแนวโน้มการพัฒนาของคลังสินค้าได้อย่างถูกต้อง
